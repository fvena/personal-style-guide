## 💡 Motivation

Consistency in code style is key to maintaining scalable and maintainable projects. While many tools like ESLint, Prettier, and Stylelint exist, configuring them repeatedly across projects can be tedious and error-prone. This package centralizes the configuration for these tools, saving time and ensuring a unified development experience across all Didor projects.

This style guide also promotes best practices.

## 🚀Getting Started

### Prerequisites

ESLint, Prettier, Stylelint, Typescript are a peer dependency, so you need to install at the root of your project:

```sh
npm install --save-dev eslint prettier stylelint typescript markdownlint
```

### Installation

To install the package, run:

```sh
npm install --save-dev personal-style-guide
```

### 🤔 Configuration Selection Guide

**Choose your configuration based on your project type:**

| Project Type         | ESLint        | Prettier        | Stylelint        | Vitest        | TypeScript                   |
| -------------------- | ------------- | --------------- | ---------------- | ------------- | ---------------------------- |
| **Node.js API**      | `eslint/node` | `prettier/node` | N/A              | `vitest/node` | `typescript/node`            |
| **Vue.js Project**   | `eslint/vue`  | `prettier/vue`  | `stylelint/vue`  | `vitest/vue`  | Use official `@vue/tsconfig` |
| **Nuxt.js Project**  | `eslint/nuxt` | `prettier/nuxt` | `stylelint/nuxt` | `vitest/nuxt` | Auto-generated by Nuxt       |
| **SCSS-only Styles** | N/A           | N/A             | `stylelint/scss` | N/A           | N/A                          |

- **[eslint-plugin-perfectionist](https://perfectionist.dev)**: Code organization rules.

- **[eslint-plugin-tsdoc](https://www.npmjs.com/package/eslint-plugin-tsdoc)**: TSDoc comment validation.
- **[eslint-plugin-eslint-comments](https://www.npmjs.com/package/@eslint-community/eslint-plugin-eslint-comments)**: ESLint comment rules.

- **[eslint-plugin-vue](https://eslint.vuejs.org)**: Vue.js specific linting rules.
- **[eslint-plugin-vuejs-accessibility](https://vue-a11y.github.io/eslint-plugin-vuejs-accessibility)**: Vue.js accessibility rules (WCAG 2.1 AA).

- **[eslint-plugin-package-json](https://github.com/JoshuaKGoldberg/eslint-plugin-package-json)**: Comprehensive package.json validation and formatting rules.

### How to Use

You need to extend one of the following configurations based on your project type:

- `personal-style-guide/eslint/node` for Node.js projects.
- `personal-style-guide/eslint/browser` for browser projects.
- `personal-style-guide/eslint/vue` for Vue.js 3 projects (includes accessibility rules).
- `personal-style-guide/eslint/nuxt` for Nuxt.js 4 projects (includes Vue and Nuxt optimizations).

Create or update `eslint.config.js` in your project root and extend the configurations you need:

```js
// Node.js projects
import eslintNode from "personal-style-guide/eslint/node";
export default [...eslintNode];

// Vue.js 3 projects
import eslintVue from "personal-style-guide/eslint/vue";
export default [...eslintVue];

// Nuxt.js 4 projects
import eslintNuxt from "personal-style-guide/eslint/nuxt";
export default [...eslintNuxt];

// Browser projects
import eslintBrowser from "personal-style-guide/eslint/browser";
export default [...eslintBrowser];
```

## Prettier

### How to Use

You can use environment-specific configurations or the default config:

**Environment-specific (recommended for new projects):**

```js
// For Node.js projects
export { default } from "personal-style-guide/prettier/node";

// For Vue.js projects
export { default } from "personal-style-guide/prettier/vue";

// For Nuxt.js projects
export { default } from "personal-style-guide/prettier/nuxt";

// For browser/frontend projects
export { default } from "personal-style-guide/prettier/browser";
```

**Default configuration (maintains backward compatibility):**

```js
export { default } from "personal-style-guide/prettier/index.js";
```

**Custom extensions:**

```js
import prettierConfig from "personal-style-guide/prettier/node";

export default {
  ...prettierConfig,
  // your custom overrides
};
```

### Why We Don't Use Prettier with ESLint or Stylelint

To improve **IDE performance** and avoid redundant processing, we separate the responsibilities of Prettier and linters (ESLint/Stylelint):

- **ESLint and Stylelint**: Focus only on code validation and enforcing rules.
- **Prettier**: Handles code formatting exclusively.

By not integrating Prettier directly into ESLint or Stylelint:

1. **Faster IDE Performance**: Linters don’t need to process formatting errors, reducing overhead.
1. **Avoid Duplicated Tasks**: Prevent running the file through both a linter and Prettier simultaneously.

## Vitest

### Features Integrated

- **Performance Optimization**: Threads vs forks configuration based on environment
- **Coverage Integration**: V8 coverage provider with intelligent reporting
- **Mock Utilities**: Common patterns and utilities for different project types
- **Environment Detection**: Automatic CI vs development optimizations
- **Setup Files**: Pre-configured mocks and test utilities

### How to Use

You can use environment-specific configurations based on your project type:

**Environment-specific configurations:**

```js
// For Node.js projects
export { default } from "personal-style-guide/vitest/node";

// For Vue.js projects
export { default } from "personal-style-guide/vitest/vue";

// For Nuxt.js projects
export { default } from "personal-style-guide/vitest/nuxt";
```

**Custom extensions:**

```js
import { defineConfig, mergeConfig } from "vitest/config";
import vitestConfig from "personal-style-guide/vitest/node";

export default mergeConfig(
  vitestConfig,
  defineConfig({
    test: {
      // your custom overrides
      testTimeout: 10000,
    },
  }),
);
```

### Configuration Features

**Performance Optimizations:**

- **Development**: 2-4 threads, coverage disabled, clean output
- **CI**: Unlimited threads, coverage enabled, detailed reports
- **Watch Mode**: Intelligent file watching with exclusions

**Built-in Utilities:**

- **Test Factories**: User, API response, and error generators
- **Common Mocks**: Fetch, localStorage, sessionStorage, console
- **Framework Specific**: Vue Test Utils, Nuxt composables, Node.js APIs

**Coverage Configuration:**
Coverage is enabled but non-blocking by default. To enforce thresholds:

```js
// In your vitest.config.js
export default mergeConfig(
  baseConfig,
  defineConfig({
    test: {
      coverage: {
        thresholds: {
          statements: 80,
          branches: 80,
          functions: 80,
          lines: 80,
        },
      },
    },
  }),
);
```

## Testing Patterns and Examples

### Common Testing Patterns

**Unit Testing with Utilities:**

```js
// tests/auth.test.js
import { describe, it, expect } from "vitest";

describe("Authentication Service", () => {
  it("validates user tokens", async () => {
    // Use global utilities (available in all configurations)
    await global.testUtils.sleep(10); // Simulate async delay

    const result = validateToken("valid-token");
    expect(result).toBe(true);

    await global.testUtils.flushPromises(); // Ensure all promises resolve
  });
});
```

**Node.js API Testing:**

```js
// tests/api.test.js
describe("API Endpoints", () => {
  beforeEach(() => {
    // Mock environment variables
    global.nodeTestUtils.mockEnv({
      NODE_ENV: "test",
      JWT_SECRET: "test-secret",
    });

    // Mock database connection
    global.nodeMocks.mockDatabase();
  });

  it("creates user successfully", async () => {
    const request = global.nodeTestUtils.createHttpRequest({
      method: "POST",
      body: { name: "John", email: "john@example.com" },
    });

    const response = await createUser(request);
    expect(response.status).toBe(201);
  });
});
```

**Vue Component Testing:**

```js
// tests/Button.test.js
import { mount } from "@vue/test-utils";
import Button from "@/components/Button.vue";

describe("Button Component", () => {
  beforeEach(() => {
    // Mock Vue Router
    global.vueMocks.mockVueRouter();
  });

  it("renders with correct props", () => {
    const wrapper = mount(Button, {
      props: { variant: "primary", disabled: false },
    });

    // Use global Vue utilities
    const button = global.vueTestUtils.findByTestId(wrapper, "button");
    expect(button.classes()).toContain("btn-primary");
  });

  it("handles click events", async () => {
    const mockRouter = global.vueTestUtils.createMockRouter([{ path: "/", component: {} }]);

    const wrapper = mount(Button, {
      global: { plugins: [mockRouter] },
    });

    await wrapper.trigger("click");
    expect(mockRouter.push).toHaveBeenCalled();
  });
});
```

**Nuxt Component Testing:**

```js

describe("Index Page", () => {
```

## TypeScript

### How to Use

This style guide provides multiple TypeScript configs. These configurations are based on the environment you are working on, so you need to extend one of the following configurations:

- `personal-style-guide/typescript/node` for Node.js projects.
- `personal-style-guide/typescript/browser` for browser projects.

Ensure you have a `tsconfig.json` in your project root. You can extend the provided configuration if necessary:

```json
// For Node.js projects
{
  "extends": "personal-style-guide/typescript/node"
}

// For browser projects
{
  "extends": "personal-style-guide/typescript/browser"
}
```

### Vue.js 3 Projects

**For Vue.js 3 projects, use the official Vue TypeScript configuration instead:**

```bash
npm install --save-dev @vue/tsconfig
```

```json
{
  "extends": "@vue/tsconfig/tsconfig.dom.json"
}
```

> **Why use the official Vue config?** Vue projects created with `create-vue` or Vite already include optimized TypeScript configurations. The official `@vue/tsconfig` (maintained by the Vue team) provides the most up-to-date and compatible settings for Vue 3 + Vite projects.

### Nuxt.js 4 Projects

**For Nuxt.js 4 projects, TypeScript configuration is automatically generated:**

Nuxt 4 automatically generates a `tsconfig.json` file with optimal settings for your project structure. No additional configuration is needed - Nuxt handles TypeScript setup automatically based on your project's needs.

> **Why no custom config?** Nuxt 4 provides automatic TypeScript configuration that adapts to your project structure, auto-imports, and dependencies. The generated configuration is always up-to-date with the latest Nuxt features and TypeScript best practices.

## 🔧 Troubleshooting

### Common Issues

**Configuration not loading:**

```bash
# Verify the import path is correct
node -e "import('personal-style-guide/eslint/node').then(console.log)"
```

**Plugin missing errors:**

- Ensure peer dependencies are installed: `npm install --save-dev eslint prettier stylelint vitest typescript`
- For Vue.js projects: `npm install --save-dev vue @vue/test-utils`
- For Nuxt.js projects: `npm install --save-dev nuxt @nuxt/eslint`
- For testing: `npm install --save-dev jsdom @vitest/coverage-v8`
- For Pug support: `npm install --save-dev @prettier/plugin-pug`

**TypeScript errors in configs:**

- Make sure you're using TypeScript 5.0+ as specified in peerDependencies
- Verify your tsconfig.json extends the correct environment config

**Vitest configuration issues:**

- Verify `vitest.config.js` imports the correct environment config
- Check that setup files are accessible (they're exported from the package)
- For Vue testing: ensure `@vue/test-utils` and `jsdom` are installed
- For Node testing: some globals might need explicit setup in your test environment

**Conflicting rules:**

- Check if you have other ESLint/Prettier configs that might conflict
- Use `npx eslint --print-config file.js` to see the final merged configuration
