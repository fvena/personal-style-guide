# ADR-010: ESLint v10 feature adoption

**Status:** Accepted
**Date:** 2026-03

## Context

ESLint v10.0.0 was released in February 2026. Several features relevant to kata were introduced across v9.22+–v10.1:

1. **`defineConfig()` with `extends`** (v9.22.0) — composable config helper.
2. **Multithreaded linting** (v9.34.0) — `--concurrency` CLI flag for parallel file processing.
3. **Bulk suppressions** (v9.24.0 CLI, v10.1.0 API) — suppress all existing violations to adopt new rules gradually.
4. **Official CSS/HTML plugins** (`@eslint/css`, `@eslint/html`) — ESLint-native CSS and HTML linting.

Additionally, v10.0.0 changed config file resolution to start from each linted file's directory rather than CWD, which benefits monorepos.

## Evaluation

### 1. defineConfig() with extends — Already adopted

kata's `eslint/base.js` already imports `{ defineConfig, globalIgnores }` from `'eslint/config'` and uses `extends` internally (e.g., `disableTypeChecked` for `.js` and `.yml` files). The root `eslint.config.js` also uses `defineConfig`.

The CLI generates consumer configs using the spread pattern (`export default [...eslintNode]`). This remains simpler than `defineConfig({ extends: [...] })` for the common case and works universally. No change needed in CLI output.

**Decision:** Already adopted internally. No change to CLI-generated configs.

### 2. Multithreaded linting — Recommend to users, don't enforce

The `--concurrency` flag (e.g., `--concurrency=auto`) provides 1.3x–3x speedups. However:

- Config values must be structured-cloneable (no functions or custom class instances). kata's configs are data-only, so this works.
- Type-aware linting with `projectService` works with multithreading.
- Vue plugin works correctly in multithreaded mode.

kata should not add `--concurrency` to the default scripts generated by the CLI — it's an optimization users can enable when they need it, and the overhead for small projects may not be worth it.

**Decision:** Document `--concurrency=auto` as a performance tip in the README troubleshooting section. Include in benchmark comparisons. Don't add to default scripts.

### 3. Bulk suppressions — Document for migration

Bulk suppressions (`eslint --suppress-all`) record existing violations into `eslint-suppressions.json`, allowing teams to adopt kata gradually without fixing all errors upfront. Only `"error"` severity rules are suppressed. v10.1.0 added API support for IDE integration.

This is valuable for teams migrating to kata on large codebases. It should be documented in the migration guides.

**Decision:** Add a section to migration guides explaining how to use `eslint --suppress-all` for gradual adoption. kata's rules are intentionally set to `"error"` for bug-catching rules, which aligns well with this feature.

### 4. Official CSS/HTML plugins — Keep Stylelint

`@eslint/css` provides basic CSS linting (no-duplicate-imports, no-empty-blocks, no-invalid-at-rules, no-invalid-properties, require-baseline). However:

- **No SCSS support** — kata's target audience uses SCSS extensively (Vue/Nuxt projects).
- **No Vue SFC `<style>` extraction** — Stylelint handles `<style scoped lang="scss">` blocks natively.
- **No property ordering** — Stylelint with `stylelint-config-recess-order` provides opinionated property ordering.
- **Complementary, not a replacement** — `@eslint/css` could catch some issues in plain CSS files, but adding another plugin increases install time and complexity for minimal benefit.

`@eslint/html` is not relevant — kata targets Node.js/TypeScript/Vue/Nuxt projects, not HTML files.

**Decision:** Keep Stylelint for CSS/SCSS/Vue styling. Do not add `@eslint/css` or `@eslint/html`. Re-evaluate if `@eslint/css` adds SCSS support.

### 5. Monorepo config resolution — Automatic benefit

v10.0.0 changed config file resolution to start from each linted file's directory. This means `eslint.config.js` in `packages/api/` is automatically found without `--config` flags. This was a pain point in v9.

**Decision:** No action needed — kata's monorepo example and guide already work correctly with v10's resolution. Document this improvement in the monorepo integration guide.

## Consequences

- **No breaking changes** — kata already runs on ESLint v10.1.0. The `peerDependencies` range (`^9.0.0 || ^10.0.0`) supports both versions.
- **Documentation updates** — Add multithreading tip to README, bulk suppressions to migration guides, and monorepo resolution note to integration guide.
- **Future consideration** — Monitor `@eslint/css` for SCSS support. If it becomes a viable Stylelint replacement, evaluate in a future ADR.
